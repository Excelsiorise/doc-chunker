# 文档导入与分块模块 · 改进建议清单

> 针对当前可运行版本（parse_document → DocumentBlock[] → chunk_blocks → Chunk[] → DocumentStore）的评审与改进列表。
> 每条包含：改什么 / 为什么改（对应考察点与题目原文）/ 为什么不进一步。

**总体判断**：骨架是对的——统一中间表示（DocumentBlock）、关系字段落在数据模型上、薄适配器、JSONL 可调试存储，都踩在正确的设计上。但对照题目原文和考察点，存在几处**要求被点名却缺席**的硬伤，和几处**核心卖点做了一半没兑现**的地方。

---

## P0：不改会直接丢分的（预算 6–8 小时）

### 1. 让搜索命中后真正"还原上下文"——目前关系字段存了但没人消费

- **改什么**：`search` 命令和 store 接口加 `--expand` 选项：命中 chunk 后沿 `prev/next_chunk_id` 取邻居窗口、沿 `heading_path` 取同节兄弟块，返回"命中块 + 扩展上下文 + 来源"的结构化结果。
- **理由**：这是最严重的问题。题目原文明确要求——"使得检索命中某个 chunk 后，能回溯到相关的扩展上下文，缓解断章取义"。当前 prev/next 和 heading_path 都存了，但检索只返回命中块本身，等于**造了钥匙没开锁**：面试官问"上下文感知体现在哪"，只能指着字段说"存了"，拿不出"用了"的演示。补上它之后，demo 里同一查询开/关 expand 的输出对比就是全场最有说服力的一页。
- **为什么不进一步**：不做 LLM 重排、不做多跳图遍历、不做 RAPTOR 摘要树——expand 是确定性的指针查找，零依赖、可单测；更复杂的还原策略收益无法在 48 小时内用评测证明，写进 roadmap。

### 2. 补上父子（small-to-big）能力——用"动态聚合"实现，不建第二层物理索引

- **改什么**：不物理生成 parent 块，而是在 store 上加 `get_section(chunk_id)` 方法：按该 chunk 的 `heading_path` 现场把同节所有 chunk 按序拼成"父上下文"返回，作为 expand 的一个档位（`--expand section`）。
- **理由**：父子是 2025–2026 生产环境采用最广、评测基准中最稳定的上下文感知模式，也是先前确定的主力策略；当前版本完全没有它，"上下文感知策略"这条答卷就只剩相邻关系一种，单薄。用动态聚合补，代码量约 30 行，因为 heading_path 已是现成的 parent 边界定义。
- **为什么不进一步**：标准双层索引（child 检索 + parent 物理存储）要维护两层数据、文档更新时双层刷新、存储翻倍；动态聚合用现有字段达到等效演示效果。设计文档写明："parent 在本实现中是 heading section 的逻辑视图而非物理块，代价是查询时多一次聚合，换来存储单一真相源"——这本身就是一条漂亮的取舍论述。

### 3. 把 store 抽成抽象接口 + 加一个 InMemory 后端 + 契约测试

- **改什么**：定义 `ChunkStore` ABC（方法从 expand 的查询需求反推：get / get_neighbors / get_by_heading / iter_all / write_document / search），现有 JSONL 实现改为它的一个后端；新增几十行的 `InMemoryStore`；用 pytest 参数化 fixture 让两个后端跑同一套契约测试。
- **理由**：题目原文写的是"设计一个 ChunkStore / DocumentStore **抽象**，至少实现一个……后端"——抽象是名词性要求，当前只有具体类，严格说这条没答完。"便于单测"这半句要求，InMemory 后端就是标准答案。
- **为什么不进一步**：不加 SQLite。JSONL 已满足"至少一个本地文件后端"，SQLite 的事务/索引/FTS5 在当前数据量下无可演示的收益；接口抽出来之后，"再给一周加 SQLite/向量库 = 新写一个类"这句话就有了着落，比赶工一个 SQLite 更值钱。

### 4. 分块不跨边界：禁止 chunk 跨越 heading 变化点和 block_type 变化点

- **改什么**：`chunk_blocks()` 的合并循环里加一条规则：当前 block 的 `heading_path` 或 `block_type` 与 buffer 中的不一致时，强制封块再开新块。
- **理由**：当前"chunk 继承首个 block 的 heading_path"，意味着一个 chunk 可以横跨两个章节，后半部分内容被**错误地标注**成前一章的——这会同时污染 expand 的同节聚合（改进 2）和来源引用，属于静默的数据正确性 bug，正是考察点"批判性思维：看得出 AI 的产出哪里错了"最爱抓的类型。附带收益：表格行 block 不会再和正文混进同一 chunk，表格原子性顺手就有。
- **为什么不进一步**：不做语义分块。实证结论明确——比基线慢约 14 倍、NAACL 2025 的分析认为其计算成本换不来稳定收益；标题和块类型是文档作者亲手给出的确定性边界信号，先把免费的结构信号用干净，才轮得到花钱的语义信号。

### 5. 补测试与一个微型对照评测

- **改什么**：
  - (a) 不变量单测：非强制切分的 chunk 以句末标点结尾、长度 ≤ max_chars、全部 chunk 拼接去重叠后无损还原原文、同输入同配置输出确定；边界样例含中英混排、句号在引号内、单句超长、空文档。
  - (b) 微型评测：手工造 10–20 个"答案藏在上下文里"的 QA 对，跑三种配置（naive 基线 / +边界规则 / +expand），出一张命中率对比小表。
- **理由**：交付物清单里"测试流程"和"测试结果或 demo"是两条独立要求；考察点第 4 条原话是"用测试、复现、对照等方法证明代码确实按预期工作，而不是看起来能跑"。对照表是整份交付物里最直接回应它的东西。
- **为什么不进一步**：不用 RAGAS、不用 LLM-as-judge——它们引入外部模型依赖和打分不确定性，20 个确定性 QA 的 exact-hit 统计在这个规模下更可信、可复现，且能在干净环境离线跑通。

## P1：高性价比、时间允许就做（预算 3–4 小时）

### 6. PDF 标题：读书签（outline），读不到就诚实声明

- **改什么**：pypdf 读 PDF 自带的书签目录，有书签时把"页码→章节"映射进该页 block 的 heading_path；无书签的 PDF 在 manifest 里记 `pdf_headings: unavailable`。
- **理由**：题目元数据要求点名了"标题"，当前 PDF 路径 heading_path 恒为空，五种格式里唯独它缺了这项。书签是确定性信号，几十行代码。
- **为什么不进一步**：不换 PyMuPDF 做字号启发式——PyMuPDF 是 AGPL 许可（商业公司环境里是真实顾虑，面试提这点非常加分），字号规则又脆弱需要按文档调参；更不上 MinerU/Docling，模型下载与环境风险违背"干净环境跑起来"的交付要求。声明"数字原生且带书签的 PDF 支持标题层级，否则退化为页级定位"是合格的边界管理。

### 7. overlap 改为句子粒度，且 expand 开启时默认 0

- **改什么**：封块时把尾部最后 N 个**完整句子**作为下一块开头（而不是按字符数回退截取）；配置上让 `expand` 与 `overlap` 互斥推荐。
- **理由**：字符级尾部重叠会在重叠区制造新的拦腰截断，恰好违反本题"避免拦腰截断"的字面要求，属于自我矛盾；overlap 和邻居扩展解决的是同一个问题，前者靠物理复制、后者靠指针，同时开是双倍成本。
- **为什么不进一步**：不做重叠去重的检索端逻辑（相邻块同时命中的去重）——那是下游检索模块的职责，写进边界声明即可。

### 8. 导出契约加固：schema_version + 引用完整性校验

- **改什么**：chunks.jsonl 的每行（或 manifest）加 `schema_version`；写一个 `validate_export()`：所有 prev/next 引用都存在、互为镜像（A.next==B ⟺ B.prev==A）、无孤儿，ingest 收尾时自动跑。
- **理由**：题目反复强调"输入输出契约"；引用完整性坏掉时 expand 会静默丢上下文，这是新增功能（改进 1）的直接护栏。约 50 行。
- **为什么不进一步**：不引入 pydantic 全面重写模型——dataclass + 手写校验器在当前规模够用，pydantic 值得做但属于"再给一周"里和 HTTP/MCP 服务化一起上的事。

### 9. doc_id 与变更检测的小修

- **改什么**：doc_id 保持 path-hash 不动，但 manifest 里补记**内容 SHA1**；重复 ingest 时内容未变可跳过（打印 "unchanged, skipped"）。"静默修正非法 overlap"改成**警告日志 + 写进 manifest**。
- **理由**：path-hash 的取舍（重命名即新文档 vs 内容寻址）要能说清，内容哈希提供了增量更新的钩子；静默修正配置是可复现性的敌人——评测时你以为跑的是 overlap=800，实际被悄悄改了，对照表就失真了。
- **为什么不进一步**：不做完整的增量更新（只重切变化的章节）——需要块级 diff，复杂度陡增，roadmap 一句话带过。

### 10. 补第三层：SKILL.md + 一个最小下游消费者脚本

- **改什么**：写 SKILL.md（何时 ingest、何时 search、expand 参数怎么选、结果里的来源字段怎么解读）；写 50 行 `demo_retriever.py`，只读 chunks.jsonl 不 import 本包，走"加载→检索→expand→带引用输出"全流程。
- **理由**：三层架构（核心包 / Tool 适配器 / SKILL.md）已有前两层，缺第三层，而 Skill 恰恰是向面试官证明"我理解 nanobot 的 Tool/Skill 二分"的最便宜证据；独立消费者脚本则是导出契约"真的可被下游消费"的可执行证明，也是 demo 主角。
- **为什么不进一步**：消费者不接 embedding——保持确定性关键词检索，理由写明：本模块边界止于"可被检索消费的结构"，语义检索是下游职责；接口上留 `VectorIndex` 的位置说明将来在哪插。

## P2：明确不做、写进设计文档"取舍与 roadmap"一节的

> 这部分不是偷懒，是题目考察点里"哪里过度设计"的正面回应，每条一句话理由即可。

| 不做的事 | 一句话理由 |
| --- | --- |
| MinerU / Docling / OCR | 环境与许可风险 > 48h 收益，解析层已留 adapter 位 |
| 语义分块 / Late Chunking / Contextual Retrieval | 分别受制于成本收益比、长上下文 embedding 模型依赖、逐块调 LLM 的入库成本；且三者与已实现策略解决的是不同层面的问题（这句辨析要写进文档） |
| SQLite / 向量数据库 | 接口已就位，插新后端即可 |
| HTTP / MCP 服务化 | 下游是本地 demo，进程内 + CLI 已覆盖两条调用路径 |
| token 计量 | 当前以字符计量并在文档中写明换算关系与截断风险，`length_function` 留作注入点——模块不该绑定某个特定 embedding 模型的 tokenizer，下游选型未定时字符是无依赖的确定性度量 |

## 执行顺序建议

P0 中的 1、2、4 三条互相咬合：边界规则保证 heading_path 干净 → section 聚合才正确 → expand 才有得演示。

**推荐顺序：4 → 2 → 1 → 3 → 5**，每做完一条把决策同步进 DECISIONS.md。

**如果时间只够做三件事：做 4、1、5**——数据正确性、核心卖点、验证方法论，这是评分权重最高的三角。
